# 九号公司 HR2.0 校招管理系统蓝图方案

## 1. 业务需求 (Business Requirements)

### 1.1 业务概述
本项目旨在构建一套集成化、自动化的校招管理平台，替换原有的 OA 发起方式，实现从校招需求发起到岗位发布的闭环管理。
*   **校招管理**：在 EHR 系统发起校招申请，通过接口对接 Moka 招聘系统，实现招聘需求（草稿）的新建、审批状态的同步；核心围绕校招需求收集、审批、变更及发布进行管理。
*   **非窗口期管理**：支持非窗口期的特殊需求申请与变更，通过严格的字段管控（如仅允许修改人数）来规范流程。

### 1.2 业务目标
1.  **流程闭环**：打通 EHR、OA、Moka 系统，实现校招需求数据不落地流转。
2.  **需求规范化**：通过标准化的申请单与变更单，规范各部门的校招需求填报，确保职族、序列、职级等信息准确。
3.  **灵活管控**：区分“窗口期”与“非窗口期”的业务逻辑，满足不同阶段的管理诉求。
4.  **数据可视化**：提供多维度的报表与监控平台，实时掌握各部门、各序列的招聘进度与审批状态。
5.  **高效协同**：明确需求部门、一级部门、公司层级的审批职责，提升协同效率。

---

## 2. 整体解决方案 (Overall Solution)

### 2.1 解决方案架构
系统以 **“参数设置 -> 需求收集 -> 公司审核 -> 岗位发布”** 为核心链路。

*   **配置端**：
    *   **参数设置**：定义招聘窗口期、招聘项目等基础配置。
*   **业务端**：
    *   **需求发起**：EHR 发起（支持批量导入/复制） -> OA 审批 -> Moka 创建职位。
    *   **变更管理**：EHR 发起变更（支持非窗口期管控） -> OA 审批 -> Moka 更新职位。
    *   **需求收集**：支持校招组、校招PM多角色视图，分层级汇总与提交。
*   **监控端**：
    *   **校招报表**：全视角的校招数据统计与明细查询。

### 2.2 方案设计要点
1.  **多角色协同**：明确区分招聘 HR、HRBP、校招 PM、SSC 在流程中的职责。
2.  **差异化流程**：
    *   **窗口期 vs 非窗口期**：非窗口期变更单仅允许修改“招聘岗位人数”，其他核心字段强制只读，确保编制严肃性。
3.  **暂无编制管控**：当前阶段校招业务暂不引入强编制校验（Headcount Control），重点在于需求的审批与发布。

### 3.1 校招申请单 (Recruitment Application)
**功能描述**：用于发起校园招聘需求的申请。

#### 3.1.1 方案设计要点
1.  **双模式支持**：系统需同时支持“窗口期批量申请”与“非窗口期单个申请”两种模式。窗口期需关联具体的校招项目，非窗口期则为独立单据。
2.  **效率工具**：提供岗位信息“批量导入”功能，支持 Excel 模板上传，减轻业务部门填报负担。
3.  **智能辅助**：在选择岗位时，系统自动关联并展示该岗位的“任职资格”信息，辅助申请人准确填报。

#### 3.1.2 业务场景
*   **场景一（窗口期）**：校招启动后，业务经理/BP 登录系统，选择“202X届春招”项目，通过下载模板批量填报本部门的 20 个校招岗位需求，上传后系统自动校验必填项，确认无误后一次性提交审批。

### 3.2 校招变更单 (Recruitment Change Order)
**功能描述**：用于对已发布的校招需求进行变更。

#### 3.2.1 方案设计要点
1.  **差异化管控**：
    *   **窗口期**：允许变更内容较多，包括招聘数量（增/减/取消）、人才类型、岗位信息、需求部门（仅限一级部门内变更）。
    *   **非窗口期**：严格管控，仅允许“招聘数量减少”或“取消需求”，不允许增加 HC 或变更岗位属性。
2.  **关联校验**：变更单必须关联已审批通过的原需求单据。

#### 3.2.2 业务场景
*   **场景一（窗口期调整）**：校招进行中，某业务线发现原定的“Java开发”需求量过大，需将部分名额调整为“前端开发”。BP 发起窗口期变更单，减少 Java 岗人数，新增前端岗人数，提交审批。
*   **场景二（非窗口期缩编）**：校招结束后，某部门因组织架构调整，需取消尚未入职的 2 个 HC。BP 发起非窗口期变更单，将对应岗位招聘人数调减为 0（即取消），系统自动锁定其他字段不可编辑。

### 3.3 校招审批单 (Recruitment Approval)
**功能描述**：各级部门查看和审批招聘需求。

#### 3.3.1 方案设计要点
1.  **多维度视图**：
    *   **需求部门**：关注本部门详情。
    *   **一级部门**：关注下辖所有需求部门的汇总与明细，支持按“序列”维度统计。
    *   **公司层级**：关注全公司汇总，支持按“一级部门”和“序列”双维度统计。
2.  **数据下钻**：支持从汇总数据点击穿透至具体的明细单据，便于审批人追溯。
3.  **高管定制视图**：为 CTO 提供专属视图，仅展示“九号星”及“技术研发”序列的数据，提升审批聚焦度。

#### 3.6.1 方案设计要点
1.  **层级汇总**：
    *   **校招 PM**：负责一级部门维度的数据把关，确认无误后统一提交给主 BP 和一级部门负责人。
    *   **校招组**：负责全公司维度的数据把关，确认无误后提交给公司高管层（CTO/HRVP/董事长）。
2.  **灵活驳回**：支持按“实际需求部门”粒度进行驳回，无需退回整个一级部门的单据，提高流转效率。

#### 3.6.2 业务场景
*   **场景**：校招 PM 在审核“研发中心”的需求时，发现“测试部”的申报人数异常，但“开发部”正常。PM 可直接勾选“测试部”的单据点击“驳回”，填写驳回原因；同时将“开发部”的单据勾选“提交”，流转至下一环节。
